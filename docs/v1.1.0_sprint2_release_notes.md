## Nuevas Características

### 1. Dynamic SLO/SLA Engine

**Files**:
- [`src/slo/dynamic_engine.rs`](../src/slo/dynamic_engine.rs)
- [`src/slo/contract_manager.rs`](../src/slo/contract_manager.rs)

**Description**:
Complete dynamic Service Level Objective evaluation engine with adaptive threshold adjustment, CPU-fallback evaluation mode, and full SLA contract lifecycle management.

**Key Features**:

- **Dynamic Threshold Adjustment**: Thresholds automatically adapt based on historical metric patterns using configurable adjustment rates, reducing false positives in stable environments.
- **CPU Fallback Mode**: When CPU load exceeds `cpu_fallback_threshold` (default 0.85), the engine switches to static evaluation mode to reduce computational overhead.
- **Multi-level Compliance**: Healthy, Warning, Breach, and Critical states with configurable warning fractions and critical multipliers.
- **Audit Hash**: Each evaluation result includes a deterministic audit hash for tamper-evident evaluation trails.
- **SLA Contract Manager**: Full contract lifecycle with binary serialization, metric validation, penalty/reward clauses, and automatic expiration handling.
- **Contract Clauses**: Penalty and reward clause types triggered by metric threshold violations.

**API**:
```rust
// Create engine with dynamic thresholds
let mut engine = DynamicSLOEngine::default_engine();

// Add evaluation rule
let rule = SLORule::new(
    "rule-1", "CPU Usage", "cpu_usage",
    80.0,      // threshold
    0.90,      // warning fraction
    1.20,      // critical multiplier
    60,        // window seconds
    3,         // max breaches
);
engine.add_rule(rule)?;

// Report metrics and evaluate
engine.report_metric("cpu_usage", 75.0);
let results = engine.evaluate_cycle()?;
```

### 2. Liquid Governance v2

**Files**:
- [`src/governance/liquid_v2.rs`](../src/governance/liquid_v2.rs)
- [`src/governance/voting_mechanism.rs`](../src/governance/voting_mechanism.rs)

**Description**:
Second-generation liquid governance system with cryptographic reputation-weighted voting, 72h time-lock for critical proposals, Sybil detection, and batch vote processing.

**Key Features**:

- **Cryptographic Reputation**: Vote weights calculated from node trust scores, staking credits, and uptime history with cryptographic signature verification.
- **72h Time-lock**: Critical proposals require 72-hour waiting period before execution, preventing rushed governance decisions.
- **Sybil Detection**: Automatic detection of Sybil clusters based on ASN, IP prefix, and voting history similarity analysis.
- **Delegation Chains**: Weight delegation with chain resolution up to configurable depth, supporting liquid democracy patterns.
- **Batch Vote Processing**: Efficient batch processing of votes with quorum calculation, tie-breaking, and abstain handling.
- **Anti-Whale Caps**: Maximum vote weight capping to prevent single-node governance dominance.

**API**:
```rust
// Register node with cryptographic profile
let profile = NodeProfileV2 {
    node_id: "node-1".to_string(),
    trust_score: 0.9,
    staking_credits: 1000.0,
    uptime_history: 0.99,
    crypto_signature: "sig-1".to_string(),
    asn: "ASN1".to_string(),
    ip_prefix: "192.168.1.0/24".to_string(),
};
gov.register_node(profile)?;

// Create and vote on proposals
let proposal = gov.create_proposal("Title", "Desc", "node-1", false)?;
gov.cast_vote("node-2", &proposal.id, true)?;
let result = gov.execute_proposal(&proposal.id)?;
```

### 3. Real-time Telemetry

**Files**:
- [`src/web/realtime.rs`](../src/web/realtime.rs)
- [`src/monitoring/streaming_metrics.rs`](../src/monitoring/streaming_metrics.rs)

**Description**:
WebSocket/SSE streaming backend for real-time telemetry events with session management, rate limiting, event filtering, and catch-up support.

**Key Features**:

- **SSE Session Management**: Create/close sessions with per-session event type subscriptions and rate limiting.
- **Event Type Filtering**: Sessions subscribe to specific event types (Metrics, Governance, System, SLO, SLA) and only receive matching events.
- **Sequential Event IDs**: Monotonically increasing sequence numbers for event ordering and catch-up support.
- **Catch-up Events**: New clients can request historical events from a specific sequence number.
- **Rate Limiting**: Per-session rate limiting with automatic window reset.
- **Streaming Metrics**: Windowed metric collection with Counter, Gauge, Histogram, and Rate aggregation types.
- **Histogram Buckets**: Automatic histogram summary generation with configurable bucket boundaries.

**API**:
```rust
// Create telemetry backend
let mut backend = TelemetryBackend::new();

// Subscribe session to event types
backend.create_session(
    "client-1".to_string(),
    vec![TelemetryEventType::Metrics, TelemetryEventType::Governance],
)?;

// Publish event (filtered to subscribed sessions)
let result = backend.publish_event(
    TelemetryEventType::Metrics,
    serde_json::json!({"cpu": 75.0}),
    Some("node-1".to_string()),
);

// Catch-up from sequence 0
let events = backend.get_catchup_events(0, &[TelemetryEventType::Metrics]);
```

## Breaking Changes

### API Signature Updates

- `DynamicSLOEngine::new()` now takes `DynamicSLOConfig` (use `default_engine()` for defaults)
- `VotingMechanism::new()` now takes `VotingConfig` (use `default_mechanism()` for defaults)
- `report_metric()`, `cast_vote()`, `validate_contract()` now take `&str` instead of `String`
- `evaluate_cycle()` now returns `Result<Vec<EvaluationResult>, DynamicSLOError>`
- `NodeProfileV2` and `ProposalV2` use struct literals (no `::new()` constructor)
- `BatchVote.decided` is now `Option<VoteDirection>` instead of `bool`
- `VotingStats.total_votes` replaces `total_voters`

## Migration Guide

### From v1.0.0 to v1.1.0-sprint2

1. **Enable feature flag**: Add `v1.1-sprint2` to your Cargo.toml features
2. **Update DynamicSLOEngine creation**: Use `default_engine()` or provide `DynamicSLOConfig`
3. **Handle Result types**: All `evaluate_cycle()` calls now need `?` or `.expect()`
4. **String to &str**: Remove `.to_string()` calls for method parameters
5. **Struct literals**: Use `{ field: value, ... }` syntax for `NodeProfileV2`
