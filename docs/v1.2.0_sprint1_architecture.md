## LP-46: Multi-Chain Registry

### Responsabilidad
Gestionar el registro, estado y salud de múltiples redes blockchain conectadas al nodo ed2kIA.

### Diseño

```
MultiChainRegistry
├── chains: HashMap<String, ChainEntry>
├── config: RegistryConfig
│   ├── max_chains: usize
│   └── health_check_interval_ms: u64
└── API
    ├── register_chain(config) → Result<(), MultiChainError>
    ├── unregister_chain(chain_id) → Result<(), MultiChainError>
    ├── update_state(chain_id, state)
    ├── get_chain(chain_id) → Option<&ChainEntry>
    ├── get_active_chains() → Vec<&ChainEntry>
    ├── health_check() → HashMap<String, ChainState>
    └── get_chains_by_protocol(protocol) → Vec<&ChainEntry>
```

### Estados de Cadena
- **Connected**: Cadena activa y operativa
- **Syncing**: Cadena sincronizando (considerada active)
- **Disconnected**: Cadena desconectada
- **Error(msg)**: Cadena en estado de error

### Decisiones de Diseño
- `get_active_chains()` incluye tanto `Connected` como `Syncing` para maximizar disponibilidad
- Health check actualiza estado a `Disconnected` si heartbeat excede timeout
- Límite configurable de cadenas máximas (`max_chains`)

## LP-47: Cross-Chain Identity

### Responsabilidad
Proporcionar identidad unificada de nodo con derivación criptográfica de claves por cadena.

### Diseño

```
CrossChainIdentity
├── node_id: String
├── signing_key: SigningKey (Ed25519)
├── chain_keys: HashMap<String, ChainKeyPair>
├── reputation: CrossChainReputation
│   ├── global_score: f32
│   ├── chain_scores: HashMap<String, f32>
│   ├── total_contributions: usize
│   └── last_updated: Instant
└── API
    ├── derive_chain_key(chain_id) → Result<String, IdentityError>
    ├── register_chain_key(chain_id, public_key)
    ├── sign_message(message) → Signature
    ├── verify_signature(chain_id, message, signature) → bool
    ├── generate_proof(chain_id) → Result<IdentityProof, IdentityError>
    ├── record_contribution(chain_id)
    └── get_global_reputation() → f32
```

### Derivación de Claves
- `derive_chain_key()` computa clave determinísticamente vía firma Ed25519 de `"node_id:chain_id"`
- La clave derivada NO se almacena automáticamente; requiere `register_chain_key()` explícito
- Esto permite derivación sin estado mutable (lectura segura concurrente)

### Identity Proof
- Contiene: node_id, chain_id, signature, public_key, timestamp
- `is_expired(max_age_ms)` verifica frescura del proof
- Firmas verificables con `verify_signature()`

## LP-48: SAE Fine-Tuning Engine

### Responsabilidad
Coordinar el ciclo de fine-tuning distribuido con scheduling de learning rate y checkpoints.

### Diseño

```
FineTuningEngine
├── config: FineTuningConfig
│   ├── learning_rate: f32
│   ├── schedule: LearningRateSchedule
│   ├── batch_size: usize
│   ├── max_epochs: usize
│   └── convergence_threshold: f32
├── state: TrainingState
│   ├── epoch: usize
│   ├── batch: usize
│   ├── current_loss: f32
│   ├── best_loss: f32
│   └── is_converged: bool
├── metrics: Vec<TrainingMetric>
└── API
    ├── start_epoch()
    ├── record_batch(loss, gradient_norm) → Result<(), TrainingError>
    ├── get_learning_rate() → f32
    ├── check_convergence() → bool
    ├── get_state() → &TrainingState
    ├── create_checkpoint() → TrainingCheckpoint
    └── restore_checkpoint(&checkpoint)
```

### Learning Rate Schedules
- **Constant**: LR fijo durante todo el training
- **CosineDecay**: `lr * (1 + cos(pi * step / max_steps)) / 2`
- **StepDecay**: `lr * decay_factor^(step / step_size)`

### Convergencia
- Se detecta cuando `current_loss < convergence_threshold`
- Training metrics se almacenan para análisis post-training

## LP-49: Gradient Aggregation v3

### Responsabilidad
Agregación FedAvg con tasas adaptativas, detección de outliers y compresión.

### Diseño

```
GradientAggregatorV3
├── config: AggregatorConfig
│   ├── compression_ratio: f32
│   ├── outlier_threshold: f32
│   ├── min_participants: usize
│   └── use_reputation_weights: bool
├── submissions: HashMap<String, GradientSubmission>
├── round: u64
├── aggregated_gradients: Option<Vec<f32>>
└── API
    ├── submit_gradient(node_id, gradient) → Result<(), AggregatorError>
    ├── aggregate() → Result<AggregationResult, AggregatorError>
    ├── detect_outliers(gradients) → Vec<usize>
    ├── compute_weighted_average(gradients, weights) → Vec<f32>
    └── reset_round()
```

### Detección de Outliers (MAD)
- Calcula mediana y MAD (Median Absolute Deviation) por dimensión
- Gradientes con valores > `outlier_threshold * MAD` se marcan como outliers
- Outliers se excluyen de la agregación

### Compresión
- Top-k sparsification: solo se mantienen los `k` valores más grandes
- `k = dimension * compression_ratio`
- Reduce bandwidth sin perder señal significativa

### Flujo de Agregación
1. `submit_gradient()` → acumula gradientes por nodo
2. `aggregate()` → detecta outliers → computa weights → promedia → comprime
3. `reset_round()` → incrementa round, limpia submissions

## LP-50: Ethical Constraint Engine

### Responsabilidad
Enforcement de restricciones éticas en gradientes durante fine-tuning.

### Diseño

```
ConstraintEngine
├── constraints: HashMap<String, EthicalConstraint>
├── violations: Vec<ConstraintViolation>
├── halted: bool
└── API
    ├── add_constraint(constraint) → Result<(), ConstraintError>
    ├── remove_constraint(id) → Result<(), ConstraintError>
    ├── validate_gradient(gradient) → Result<(), ConstraintViolation>
    ├── correct_gradient(gradient) → Result<(), ConstraintViolation>
    ├── should_halt() → bool
    ├── clear_violations()
    └── get_violations() → &[ConstraintViolation]
```

### Tipos de Restricción
| Tipo | Descripción | Acción |
|------|-------------|--------|
| ValueBound | Límite min/max por feature | Clamp o reject |
| FeatureMask | Features que deben ser cero | Zero-out o reject |
| AlignmentThreshold | Mínimo de alignment score | Reject si bajo |
| DivergenceLimit | Máximo de divergencia | Scale-down o reject |

### Severidades
- **Warning**: Solo registra violación
- **Correction**: Corrige gradient automáticamente
- **Critical**: Detiene training (`halted = true`)

### Flujo de Validación
1. `validate_gradient()` → verifica todas las constraints → retorna primera violación
2. `correct_gradient()` → aplica correcciones (clamp, mask, scale)
3. `should_halt()` → retorna true si hay violación Critical

## Integración entre Módulos

### Pipeline Completo
```
1. Registry: register_chain() → gestionar conexiones blockchain
2. Identity: derive_chain_key() → obtener identidad por cadena
3. Training: FineTuningEngine → entrenar localmente
4. Aggregation: submit_gradient() → enviar gradientes
5. Aggregation: aggregate() → combinar gradientes
6. Constraints: validate_gradient() → verificar ética
7. Constraints: correct_gradient() → corregir si necesario
```

### Datos Compartidos
- `CrossChainIdentity.reputation` → weights en `GradientAggregatorV3`
- `MultiChainRegistry.get_active_chains()` → cadenas disponibles para identity
- `FineTuningEngine.create_checkpoint()` → state para gradient extraction
- `ConstraintEngine.should_halt()` → control de flujo del pipeline
