## 2. LP-98: Halo2 Integration

### 2.1 Overview
Replace hash-based proof simulation in `src/zkp/async_zkp_v5.rs` with Halo2 backend.

### 2.2 Architecture

```
┌─────────────────────────┐
│   AsyncZKPV5 (Facade)   │
│   - VRF Sampling        │
│   - Batch Management    │
│   - Circuit Selection   │
└───────────┬─────────────┘
            │
     ┌──────▼───────┐
     │ ZKPBackend    │  ← New Trait
     │ (trait)       │
     └──────┬───────┘
            │
     ┌──────▼───────┐     ┌──────────────┐
     │ Halo2Backend  │     │ HashBackend  │
     │ (production)  │     │ (simulation) │
     └──────────────┘     └──────────────┘
```

### 2.3 API Design

```rust
/// Trait abstracting ZKP backend implementation.
pub trait ZKPBackend: Send + Sync {
    /// Generate a proof for the given statement.
    fn generate_proof(&self, statement: &ZKPStatement) -> Result<ZKPProof, ZKPV5Error>;

    /// Verify a proof against its statement.
    fn verify_proof(&self, proof: &ZKPProof, statement: &ZKPStatement) -> Result<bool, ZKPV5Error>;

    /// Get backend name for metrics/logging.
    fn name(&self) -> &str;
}

/// Halo2-based backend (feature-gated).
#[cfg(feature = "halo2")]
pub struct Halo2Backend {
    params: ParamsProver,
    circuit: ZKPCircuit,
}

/// Hash-based simulation backend (default).
pub struct HashBackend;
```

### 2.4 Feature Flags

```toml
[features]
stable = ["v1.3-sprint1", "v1.3-sprint2", "v1.3-sprint3"]
halo2 = ["dep:halo2", "dep:halo2curves"]
production = ["stable", "halo2", "tokio-rt", "lz4-real"]
```

### 2.5 Deliverables
- [ ] `ZKPBackend` trait in `src/zkp/backend.rs`
- [ ] `Halo2Backend` implementation
- [ ] `HashBackend` (migrated from current code)
- [ ] Backend selection in `ZKPV5Config`
- [ ] Unit tests for both backends
- [ ] Benchmark comparison (hash vs halo2)

### 2.6 Acceptance Criteria
- Halo2 proofs verify correctly.
- Hash backend produces identical API surface.
- Feature flag `halo2` enables production backend.
- No regression in simulated mode.

## 4. LP-100: LZ4 Real Compression

### 4.1 Overview
Replace simulated LZ4 compression with actual `lz4` crate.

### 4.2 API Changes

```rust
// Before (simulated)
fn simulate_lz4_compression(data: &[Vec<f32>]) -> usize {
    (data.len() as f64 * 0.294) as usize  // Simulated 3.4x ratio
}

// After (real)
use lz4::Lz4Compressor;
pub fn compress_checkpoint(data: &[Vec<f32>]) -> Result<Vec<u8>, FineTuningV3Error> {
    let bytes = serialize(data)?;
    Ok(lz4_compress(&bytes)?)
}

pub fn decompress_checkpoint(data: &[u8]) -> Result<Vec<Vec<f32>>, FineTuningV3Error> {
    let bytes = lz4_decompress(data)?;
    Ok(deserialize(&bytes)?)
}
```

### 4.3 Deliverables
- [ ] `lz4` dependency (feature-gated as `lz4-real`)
- [ ] `compress_checkpoint()` / `decompress_checkpoint()` functions
- [ ] Update `Checkpoint` struct to store compressed data
- [ ] Compression ratio metrics
- [ ] Fallback to simulated if feature not enabled

### 4.4 Acceptance Criteria
- Real compression achieves ≥3.0x ratio on typical checkpoints.
- Decompression produces identical data (round-trip test).
- Feature flag controls real vs simulated.

## 7. Migration Plan

### 7.1 Backward Compatibility
- Simulated backends remain default (no feature flags).
- Production features opt-in via feature flags.
- API surface unchanged for existing consumers.

### 7.2 Feature Rollout
1. **Phase 1:** Hash backend + simulated (current v1.3.0).
2. **Phase 2:** Halo2 backend available via `--features halo2`.
3. **Phase 3:** Tokio async available via `--features tokio-rt`.
4. **Phase 4:** Full production via `--features production`.

## 8. Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Halo2 breaking API changes | Abstract behind `ZKPBackend` trait |
| Async migration complexity | Incremental, feature-gated |
| Performance regression | Benchmark gates in CI |
| Dependency bloat | Feature flags keep default lean |
